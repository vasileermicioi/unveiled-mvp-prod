## Why

With `@unveiled/api` packaged and able to run as its own Cloudflare Worker (change 02), the Astro app should no longer own HTTP endpoints under `/api/*` — every `/api/*` request must be forwarded to the dedicated API Worker. The Astro app keeps SSR pages, Astro Actions, session-cookie verification, and React islands, but its `/api/*` endpoint surface becomes a thin delegation. Co-locating the API Worker as a service binding on the Astro app's Worker preserves the single-origin behavior, keeps session cookies / CORS / CSP unchanged, and avoids introducing a second public hostname. The standalone (`api.unveiled.app`) deployment shape is deferred to change 06 (routing orchestrator).

## What Changes

- Add a Cloudflare service binding to the root `wrangler.toml` so the Astro app Worker can dispatch any `/api/*` request into the dedicated API Worker without crossing a public origin: `[[services]] binding = "API" service = "unveiled-api" entrypoint = "fetch"`.
- Add an early-return `/api/*` short-circuit in `src/middleware.ts` (and in the entry Worker shape generated by `@astrojs/cloudflare`) that forwards the inbound `Request` to `env.API.fetch(request)` and returns the resulting `Response` unchanged.
- Align `wrangler.api.toml` (from change 02) with the shared `SESSION` KV namespace id and `ASSETS_BUCKET` R2 binding declared by the root `wrangler.toml` so service-bound calls succeed in dev, preview, and production.
- Update `scripts/check-architecture-drift.ts` to recognize that the API container is now a separate Cloudflare Worker service binding rather than an Astro endpoint container.
- Update `package.json` (root) so `bun run preview:cloudflare` and `bun run deploy:cloudflare` chain `wrangler deploy --config wrangler.api.toml` for the API Worker before `wrangler deploy --config dist/server/wrangler.json` for the Astro app.
- Delete `src/pages/api/**` (the shims installed by change 02) once integration parity passes — no `/api/*` Astro endpoints remain after this change.
- Add coverage in `tests/api/parity.spec.ts` (status / headers / body byte-equivalence) and `tests/api/cookie-forwarding.test.ts` (cookie + streaming behavior across the service binding).

## Capabilities

### New Capabilities

_None net-new._ This change composes existing capabilities (`api-package`, `openapi-contract`, `auth`, `forms-actions`, `routing`) by moving ownership of the HTTP surface from the Astro app to the API Worker without introducing new observable behavior.

### Modified Capabilities

- `api-package`: the `/api/*` surface is now reached exclusively through the API Worker. The Astro catch-all shim from change 02 is removed; the only `/api/*` path inside the Astro app is the middleware short-circuit that delegates to the service binding.
- `openapi-contract`: the canonical OpenAPI document at `GET /api/openapi.json` is served by the API Worker via the service binding; the Astro app no longer mounts an OpenAPI endpoint.
- `auth`: the Astro app no longer mounts Better Auth's HTTP routes. Sign-in / sign-up / sign-out / password-recovery / account flows all reach the API Worker via the service binding; the Astro app keeps session-cookie verification only for SSR pages.
- `forms-actions`: Astro Actions remain the only mutation surface from the Astro app. Their canonical HTTP shape lives in the API Worker (per change 02); the Astro action wrappers in `src/actions/index.ts` keep their convenience methods and continue to call into `@unveiled/api`'s exported handlers.
- `routing`: every URL prefixed `/api/*` is dispatched to the API Worker via Cloudflare service binding; everything else continues to be served by the Astro app (or, after change 06, by the routing orchestrator).

## Impact

- **New files:** _none net-new_ (the service binding is a `wrangler.toml` change; tests live alongside existing `tests/api/`).
- **Modified files:**
  - `wrangler.toml` — adds the `[[services]] binding = "API" service = "unveiled-api" entrypoint = "fetch"` block.
  - `wrangler.api.toml` (from change 02) — adds the shared `SESSION` KV namespace id and `ASSETS_BUCKET` R2 binding.
  - `src/middleware.ts` — adds the `/api/*` short-circuit before the existing guard chain (language resolution → viewer hydration → route-table match → permission check → redirect-or-render).
  - `scripts/check-architecture-drift.ts` — updates container expectations so the API container is recognized as a service binding, not an Astro endpoint container.
  - `package.json` (root) — `preview:cloudflare` and `deploy:cloudflare` scripts chain the two Wrangler deploys.
  - `openspec/config.yaml` / `openspec/specs/**` — updated capability deltas per the Modified Capabilities list above.
- **Removed files:** `src/pages/api/**` (every file deleted after the shim window closes).
- **Dependencies changed:** _none._
- **Risks:**
  - **Service binding latency.** Adding one Cloudflare-internal hop for `/api/*` could regress p95 latency. Mitigation: measure with `wrangler tail` before/after; the hop is in-region and historically negligible; a regression test asserts parity on response body, status, and headers.
  - **Cookie forwarding.** The service-bound request must preserve the inbound `Cookie` header and the response must preserve `Set-Cookie`. Mitigation: `tests/api/cookie-forwarding.test.ts` asserts cookie parity across the binding for sign-in, sign-out, and a request that mutates session state.
  - **Streaming responses.** Stripe webhook handler and any future SSE endpoints must stream end-to-end through the service binding. Mitigation: use `ReadableStream` end-to-end and add a streaming integration test that asserts chunked transfer behavior.
  - **Better Auth session domain.** The session cookie must be readable by both the Astro app and the API Worker. Mitigation: `AUTH_COOKIE_DOMAIN` is already set per change 02; verified by `tests/api/cookie-forwarding.test.ts`.
