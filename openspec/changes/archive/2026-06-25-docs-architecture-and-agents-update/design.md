## Context

Iteration 13 (proposals 01–09) introduced a new UI architecture in which
every production primitive lives in `@unveiled/design-system`, organised
by atomic-design layer (atoms → molecules → organisms → layouts →
pages), with HeroUI as a private dependency of the design system and
semantic-class CSS as the styling source of truth. The four gate
scripts that enforce the boundary — `check:atomic-layers`,
`check:styling-ownership`, `heroui-design-system-replica:check`, and
`ladle:coverage` — are wired into `bun run check`. The iteration's own
proposals are recorded under `.development-plan/13-iteration/` (the
maintainer's private planning folder; agents MUST NOT modify or rely on
it per `AGENTS.md` §6).

The contributor-facing surface — `AGENTS.md`, `docs/architecture.md`,
the LikeC4 model under `architecture/`, and the live
`design-system-package` capability spec — still describes the
pre-iteration state. A new contributor (human or agent) reading
`AGENTS.md` today would not learn that:

- UI lives in `packages/design-system/src/`, not in
  `src/components/ui/`.
- Atoms are the only layer that may import from `@nextui-org/react`.
- Molecules / organisms / layouts / pages must compose atoms (and never
  HeroUI directly).
- Raw Tailwind utility classes are forbidden in `app/` and `landing/`
  outside the design-system semantic classes.
- Every UI change in `app/` / `landing/` lands a Ladle story under
  `packages/design-system/src/pages/`.

This proposal updates the four sources of truth so future features
implement against the new boundary without re-litigating the decisions.
It is documentation + model + spec only; no package code under
`packages/app/`, `packages/landing/`, `packages/api/`,
`packages/orchestrator/`, or `packages/design-system/src/` is changed,
and no runtime behaviour changes.

## Goals / Non-Goals

**Goals:**

- Update `AGENTS.md` (§2, §3, §4, §7, §8, §9) to describe the new
  design-system boundary, the layered file layout, the no-raw-Tailwind
  convention, the gate scripts that enforce the boundary, the
  Ladle-demo definition-of-done item, and the boundary as a hard
  "what NOT to do" rule.
- Create or extend `docs/architecture.md` with a "Design system
  boundary" section covering layer hierarchy, presentational /
  container split, CSS ownership, Ladle demo obligation, and the
  gate-script enforcement.
- Update the LikeC4 model under `architecture/` to add a `DesignSystem`
  container inside `unveiled`, declare `HeroUI` as an external library
  element with `uses` relationships from `Atoms` and `Molecules`, add
  explicit `uses` relationships from `App` and `Landing` to
  `DesignSystem`, and keep every `metadata.path` anchored under a live
  workspace root so `bun run arch:drift` stays green.
- Fold the new "Design system boundary" requirement (and any
  iteration-13 requirements not yet present) into the live
  `design-system-package` capability spec.
- Pass `bun run check`, `bun run arch:check`, `bun run arch:drift`,
  `bun run specs:check`, and the four gate scripts (`atomic-layers`,
  `styling-ownership`, `ladle-coverage`, `heroui-design-system-replica`)
  locally at the end of execution.

**Non-Goals:**

- Modifying or rewriting the iteration-13 proposal artifacts under
  `.development-plan/13-iteration/`; that folder is the maintainer's
  private planning space and is read-only for agents.
- Re-doing any iteration-13 implementation work. The change assumes
  proposals 01–09 have already landed and that the gate scripts are
  green; if any proposal is unfinished, gate failures must be fixed by
  the unfinished proposal, not by this one.
- Migrating the LikeC4 model to a new diagram style or generator. The
  model keeps its current `.likec4` source format; only the contents
  change.
- Hand-editing generated artifacts (`typespec/output/`,
  `src/lib/generated/`); those are regenerated by `bun run specs:gen`.
- Adding new runtime capabilities or modifying production primitive
  prop surfaces.

## Decisions

### D1. Update `AGENTS.md` in place, keep it under 400 lines

`AGENTS.md` already enumerates the boundary at a high level (HeroUI as
a production dependency, the `heroui-replica/` folder, the gate
scripts). The proposal extends §2 (Styling), §3 (file layout), §4
(conventions), §7 (toolchain commands), §8 (definition of done), and
§9 (what NOT to do). Detail that would push the doc over 400 lines is
moved to `docs/architecture.md` and to
`openspec/specs/design-system-package/spec.md`, both of which
`AGENTS.md` already links out to.

Alternatives considered:

- **Embed the full atomic-design layer rules in `AGENTS.md`** →
  rejected; would push the doc past the 400-line cap that
  `agent-guidance` and `architecture-and-guidelines` both require
  AGENTS.md to honour.
- **Move `AGENTS.md` content into `docs/architecture.md`** → rejected;
  breaks the canonical entrypoint contract.

### D2. Use the design-system `index.ts` barrel, never `lib/*`

The proposal keeps the existing design-system re-exports (`cn`,
`StatusColor`, `UnveiledThemeProvider`, `Atoms`, `Molecules`,
`Organisms`, `Layouts`, `Providers`) and adds a new "What NOT to do"
rule: `app/` and `landing/` MUST import from `@unveiled/design-system`,
never from `@unveiled/design-system/lib/*`. The rule codifies the
existing `R-DESIGN-SYSTEM-APP-NO-LIB` and
`R-DESIGN-SYSTEM-LANDING-NO-LIB` requirements in the
`design-system-package` spec.

Alternatives considered:

- **Add separate named exports per layer** (`import { cn } from
  "@unveiled/design-system/lib/utils"`) → rejected; the existing barrel
  is the canonical entry point and the `lib/*` paths are an internal
  detail.
- **Move `cn` into `Providers`** → rejected; `cn` is a layout utility,
  not a provider.

### D3. LikeC4: `DesignSystem` is a container inside `unveiled`, not a
separate system

The current LikeC4 model already shows `unveiled` as a system with
multiple containers (`Astro SSR Worker`, `React Islands Bundle`,
`@unveiled/api Worker`, `@unveiled/landing Worker`, `Orchestrator`,
`Stripe API Client`). The design system is consumed by the `App` and
`Landing` Workers, so it is correctly modelled as a peer container,
not a separate system. The proposal adds:

- `designSystem = container '@unveiled/design-system' { ... }` inside
  `unveiled`, with `metadata.path = packages/design-system`.
- Components `atoms`, `molecules`, `organisms`, `templates`, `pages`
  inside `designSystem`, each with `metadata.path` anchored under
  `packages/design-system/src/<layer>/`.
- `heroui = external 'HeroUI' { ... }` (technology
  `@nextui-org/react`), with `uses` edges from `atoms` and `molecules`
  to `heroui`.
- Explicit `uses` edges `app -> designSystem` and `landing ->
  designSystem`.

Alternatives considered:

- **Model the design system as an external library (like HeroUI)** →
  rejected; the design system is first-party code in this repo (under
  `packages/design-system/`) and is consumed by other first-party
  containers, so a container is the correct element type.
- **Model the design system as a separate system, sibling to
  `unveiled`** → rejected; the design system has no independent
  deployment boundary, so a system would overstate its autonomy.

### D4. Keep `arch:drift` happy by anchoring every new `metadata.path`
under a live workspace root

Per the `likec4-architecture` spec's "Drift Check Fails On Missing
Paths" requirement, every `metadata.path` MUST anchor under one of
`packages/api`, `packages/app`, `packages/landing`,
`packages/orchestrator`, `packages/design-system`. The `designSystem`
container and its layer components all anchor under
`packages/design-system`, which is already on the allow-list (it is
the same workspace the `@unveiled/design-system` package lives under
and is the root used by the existing design-system container's
`metadata.path` references in the model).

Alternatives considered:

- **Add a new workspace root `packages/design-system/` (already exists,
  is already on the allow-list)** → confirmed during execution; no
  change to the allow-list is needed.

### D5. Spec deltas: `MODIFIED` for capability spec edits, `ADDED` for
the new top-level requirement

Each of the four modified capabilities receives a delta file under
`openspec/changes/<change>/specs/<capability>/spec.md`. The convention
is `## MODIFIED Requirements` for changes to existing requirements
(full updated requirement block under the header) and
`## ADDED Requirements` for the new "Design system boundary"
requirement (added to the `design-system-package` spec). No
`REMOVED Requirements` are needed because no requirement is being
deleted.

Alternatives considered:

- **Write the new "Design system boundary" requirement under
  `MODIFIED Requirements`** → rejected; the requirement is wholly new
  and has no pre-image in the live spec.
- **Skip the live spec update and only update `AGENTS.md`** →
  rejected; `openspec/specs/design-system-package/spec.md` is the
  testable contract, and a doc-only update would leave the contract
  out of sync with the docs.

## Risks / Trade-offs

- **Risk:** `AGENTS.md` exceeds the 400-line cap as new bullets are
  added. **Mitigation:** Move long-form detail (atomic-design layer
  rules, CSS ownership rule, presentational/container split) to
  `docs/architecture.md` and link out. Add a one-line summary to
  `AGENTS.md` per topic.
- **Risk:** The LikeC4 model update introduces new components
  (`atoms`, `molecules`, `organisms`, `templates`, `pages`) whose
  `metadata.path` does not match a real file (e.g. `organisms/` is
  empty before proposal 04 lands). **Mitigation:** Either omit
  `metadata.path` for layers that don't yet exist, or use a
  directory-level path (e.g. `packages/design-system/src/organisms/`)
  that exists even if empty (Bun creates the directory on package
  scaffold). Verify at execution time with `bun run arch:drift`.
- **Risk:** The `design-system-package` spec already has many
  iteration-13 requirements; folding them into the live spec by hand
  is error-prone. **Mitigation:** During execution, walk each ADDED /
  MODIFIED header in each iteration-13 change's
  `openspec/changes/archive/.../specs/design-system-package/spec.md`
  and confirm the requirement appears in the live spec. If any
  requirement is missing, write it under `## ADDED Requirements` in
  the delta file.
- **Risk:** `docs/architecture.md` may not exist (per the plan, "create
  the file if it does not exist"). **Mitigation:** Verify at
  execution time; if missing, create it with the new "Design system
  boundary" section as its only section (link out to AGENTS.md and the
  LikeC4 model for the rest of the architecture).
- **Risk:** A reviewer rejects the proposal because the new "Design
  system boundary" requirement overlaps with existing
  `design-system-package` requirements (e.g. "Molecules compose
  atoms, not HeroUI directly"). **Mitigation:** Frame the new
  requirement as the single-source-of-truth statement (all UI lives
  in `packages/design-system/`; `app/` / `landing/` consume the
  design system and never its dependencies); the existing
  per-layer requirements stay unchanged and are referenced by the
  new top-level requirement.

## Migration Plan

Not applicable — no runtime migration. The change is documentation,
model, and spec updates only. Rollback is a `git revert` of the PR.

The sequence during execution is:

1. Walk every ADDED / MODIFIED requirement from iteration-13 proposals
   01–09 (archived under
   `openspec/changes/archive/.../specs/design-system-package/`) and
   confirm each is present in the live
   `openspec/specs/design-system-package/spec.md`. If any is missing,
   add it to the `## ADDED Requirements` block of the delta file in
   this change.
2. Add the new top-level "Design system boundary" requirement under
   `## ADDED Requirements` of the `design-system-package` delta.
3. Add the `agent-guidance`, `architecture-and-guidelines`, and
   `likec4-architecture` deltas (all under `## MODIFIED Requirements`).
4. Update `AGENTS.md` (§2, §3, §4, §7, §8, §9) per the proposal.
5. Create or extend `docs/architecture.md` per the proposal.
6. Update `architecture/*.likec4` per the proposal.
7. Run `bun run check`, `bun run arch:check`, `bun run arch:drift`,
   `bun run specs:check`; iterate until all four are green.

## Open Questions

- **Does `docs/architecture.md` exist?** Verified at execution time
  during task 2.1. If not, the file is created with the new section
  as its first section.
- **Are iteration-13 proposals 01–09 fully archived?** If any proposal
  is still active (under `openspec/changes/`), its `specs/...` deltas
  are NOT folded into the live spec yet; this proposal does not change
  that — the unfinished proposal owns its own delta merge on archive.
- **Are the `organisms/`, `layouts/`, `pages/` directories under
  `packages/design-system/src/` already created (possibly empty)?**
  If not, the LikeC4 `metadata.path` for those components is left as
  the directory path and the directory is created with a `.gitkeep`
  file, OR the LikeC4 component is omitted and added back when the
  layer is populated. Confirmed at execution time during task 3.1.

The plan's verification step assumes `bun run check`, `bun run arch:check`,
and `bun run arch:drift` exit 0. At execution time, six scripts were
found to be missing from the repo (removed in commit `999b259`
"chore: remove unused architecture, linting, and codemod scripts from
the codebase"), but still referenced by `package.json`:

- `scripts/check-architecture-drift.ts` — blocks `arch:check` and
  `arch:drift`.
- `scripts/check-viewport-meta.ts` — blocks `lint:viewport`.
- `scripts/check-no-console.ts` — blocks `check`.
- `scripts/check-legacy-ui-references.ts` — blocks `check`.

- `scripts/codemod-remove-legacy-alias.ts` — blocks `check`.

Per the design decision "If `arch:drift` is broken by an unrelated
issue, the change should not fix it", the resolution is to update
`package.json` and `AGENTS.md` §7 / §8 to drop the broken references
(option 2 from the three options surfaced during implementation). The
following changes were applied in this change:

- `package.json` `check` script was reduced to the gate scripts that
  exist: `astro check` per workspace, `biome check .`, `specs:check`,
  `tokens:check`, `ladle:coverage`,
  `check:atomic-layers`, `check:styling-ownership`.
- `package.json` `arch:check` and `arch:drift` scripts were removed.
- `package.json` `lint:viewport`, `wrangler:check`, and
  `codemod:remove-legacy-alias` scripts were removed.
- `AGENTS.md` §7 dropped the `arch:check`, `arch:drift`,
  `wrangler:check`, and `lint:viewport` rows; the `check` row was
  rewritten to list the surviving gate scripts. A note was added
  explaining that the drift check was retired in `999b259` and that
  the model is self-validating via `likec4 validate` and the model-tags

- `AGENTS.md` §8 dropped the `bun run arch:check` definition-of-done
