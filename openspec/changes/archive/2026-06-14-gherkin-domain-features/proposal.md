## Why

The repository currently has a single Gherkin feature file (`tests/features/core-platform.feature`) wired to Playwright via `tests/parity/e2e-gherkin.spec.ts` and `tests/steps/step-definitions.ts`. It is a smoke test of legacy parity, not a domain-driven specification: it mixes guest/member/partner/admin scenarios, hard-codes fixture emails, and uses `getByText` / `getByRole` selectors that break on every copy change.

In 09-iteration every feature must be backed by a gherkin spec executed as a Playwright interaction test against the real route *or* a Storybook iframe. To make that sustainable we need one feature file per domain (the 9 epics in `docs/epics.md` → 9 domains × N surfaces), a shared step-definition layer with a tight selector discipline (proximity + layout only, no `data-testid`, no copy-chained text), typed step arguments that import the TypeSpec-generated types and the design tokens, and a runner that is independent of the legacy parity suite and is ready to be extended by 09-iteration to open a Storybook iframe for component-level scenarios.

## What Changes

- Create `tests/features/<domain>/<surface>.feature` for every epic × surface:
  - `tests/features/core-platform/app-shell.feature` (header, nav, drawer, language toggle)
  - `tests/features/core-platform/discovery.feature` (filters, map, grid, empty state, pagination)
  - `tests/features/core-platform/responsive.feature` (mobile drawer, breakpoints, skeleton loaders)
  - `tests/features/identity/session.feature` (signup, login, logout, password recovery, redirects)
  - `tests/features/identity/authorization.feature` (role-gated routes, deep-link preservation)
  - `tests/features/discovery/search-filter.feature`
  - `tests/features/discovery/calendar.feature` (`.ics` download)
  - `tests/features/billing/subscriptions.feature` (Stripe checkout, webhook, portal)
  - `tests/features/billing/credits.feature` (refill, debit, admin freeze/unfreeze, ledger)
  - `tests/features/bookings/booking.feature` (book, waitlist, cancel, redeem)
  - `tests/features/bookings/redemption.feature` (voucher, secret code, URL)
  - `tests/features/operations/partner-check-in.feature`
  - `tests/features/operations/admin-crud.feature` (events, partners, members, metrics)
  - `tests/features/operations/exports.feature` (partner + admin CSV/Excel)
  - `tests/features/media/upload.feature` (R2 + SafeImage)
  - `tests/features/jobs/scheduled-emails.feature`
  - `tests/features/jobs/cron-workers.feature`
  - `tests/features/infrastructure/health-readiness.feature`
  - `tests/features/infrastructure/parity-smoke.feature` (the one happy-path scenario that lives in 09-iteration)
- Create `tests/steps/verbs/` — one file per verb (`auth.steps.ts`, `navigation.steps.ts`, `forms.steps.ts`, `lists.steps.ts`, `modals.steps.ts`, `visual.steps.ts`, `network.steps.ts`, `data.steps.ts`, `i18n.steps.ts`, `time.steps.ts`).
- Each verb file registers a small DSL importing `Given` / `When` / `Then` from a shared `tests/steps/dsl.ts`.
- The DSL enforces two selector strategies (no third):
  - **Proximity**: `getFieldNearestTo(label)`, `getButtonInside(landmark, name)`, `getLinkNearestTo(label)`.
  - **Layout**: `getByRole(...)`, `getByLabel(...)`, `getByText(...)` (with `exact: true` only when the literal text matters), and `locator("article, section, nav, main, header, footer, dialog")` filtered by the smallest containing landmark.
- Add `tests/steps/selectors/proximity.ts` and `tests/steps/selectors/layout.ts` shared modules exporting the helpers above.
- Add `tests/steps/seed.ts` that reads the fixture emails and seed data from the parity test setup so feature files do not hard-code any state.
- Add a runner `tests/parity/gherkin.spec.ts` (renamed from `e2e-gherkin.spec.ts`) that walks every `tests/features/**/*.feature` file and dispatches to the step registry. 09-iteration extends this runner to call from a Storybook iframe (see the `infrastructure/storybook-runner-extension` row in `09-iteration/01-review-existing-features.md`).
- Add a `@story(component=…, story=…)` tag schema to the runner. When a scenario carries the tag, the runner opens the Storybook iframe at the corresponding story URL and runs the same step dispatcher inside it. When the tag is missing, the runner falls back to the real route (with a warning) so legacy scenarios keep working. The Storybook story itself is authored per-feature in 09-iteration.
- Typed step arguments import the same types generated by the TypeSpec proposal (`openapi-contract` capability) and consume the design tokens (`design-tokens` capability) for assertions on class names.
- Add `openspec/specs/gherkin-domain-features/spec.md` (new capability) with the rules for adding a feature file and the selector discipline.
- Update `openspec/specs/e2e-gherkin-playwright/spec.md` to point at the new per-domain files and the shared step registry.

## Capabilities

### New Capabilities

- `gherkin-domain-features`: A complete Gherkin specification of the Unveiled MVP organized as one feature file per domain (9 epics × surfaces), backed by a shared step-definition layer that uses ONLY proximity (label-relative) and layout (semantic landmarks, ARIA roles) selectors. The features are the executable specification consumed by 09-iteration's Storybook interaction tests, with a `@story(component=…, story=…)` tag schema that lets a scenario target a Storybook iframe (real route is the fallback).

### Modified Capabilities

- `e2e-gherkin-playwright`: The single legacy `core-platform.feature` is split into per-domain feature files under `tests/features/<domain>/`. The runner in `tests/parity/e2e-gherkin.spec.ts` is renamed to `tests/parity/gherkin.spec.ts` and dispatches to the shared step registry.
- `viewer-session`: MUST have at least one gherkin scenario that exercises the viewer hydration matrix; the scenario id is referenced from the capability spec.
- `routing`: MUST have at least one gherkin scenario that exercises the `/[lang]/...` route table (public, member, partner, admin surfaces); the scenario id is referenced from the capability spec.
- `i18n-copy`: MUST have at least one gherkin scenario that exercises the language toggle and the missing-key fallback; the scenario id is referenced from the capability spec.
- `data-access`: MUST have at least one gherkin scenario that exercises a surface query through the data-access layer; the scenario id is referenced from the capability spec.
- `forms-actions`: MUST have at least one gherkin scenario that exercises a form submission (action result envelope, field errors, invalidation); the scenario id is referenced from the capability spec.

## Impact

- New files: ~18 `.feature` files under `tests/features/<domain>/`, ~10 step files under `tests/steps/verbs/`, `tests/steps/selectors/proximity.ts`, `tests/steps/selectors/layout.ts`, `tests/steps/dsl.ts`, `tests/steps/seed.ts`, `tests/parity/gherkin.spec.ts`, `openspec/specs/gherkin-domain-features/spec.md`.
- Modified files: `tests/parity/e2e-gherkin.spec.ts` (renamed + slimmed down), `tests/steps/step-definitions.ts` (split into verbs, kept as a barrel re-export), `openspec/specs/e2e-gherkin-playwright/spec.md` (MODIFIED Requirements), plus a `## MODIFIED Requirements` block in each of the five `viewer-session` / `routing` / `i18n-copy` / `data-access` / `forms-actions` capability specs.
- Removed: the legacy `core-platform.feature` content that is now covered by per-domain files. The file is kept (empty `Feature:` header) until 09-iteration migrates the last smoke scenario to `infrastructure/parity-smoke.feature`.
- No runtime, no database, no UI primitive changes. The feature files are the spec; the Playwright runner executes them; the Storybook interaction runner (09-iteration) reuses the same steps with a different `page` (Storybook iframe).
- Step arguments consume the TypeSpec-generated types (`openapi-contract` capability) and the design tokens (`design-tokens` capability); neither of those contracts is changed by this proposal, but the new step registry becomes a downstream consumer of both.
