## Context

The current HTTP surface is a flat set of Astro endpoint files under `src/pages/api/**`. They are convenient but coupling them to the Astro runtime blocks three things we need soon:

1. Running the API independently of the Astro app (Workers, cron jobs, integration tests).
2. Sharing the same TypeSpec-derived Zod validators across the HTTP boundary, cron jobs, and any future client SDK.
3. Moving to a thin orchestrator Worker that dispatches `/api/*` to one runtime and `/*` to the Astro app.

The 12-iteration plan also needs a deployable Worker shape (`packages/api/dist/worker.js`) consumed by a separate `wrangler.api.toml`. Today `packages/api/` exists only as an empty workspace folder declared in the root `package.json` and `packages/README.md`. The work in this change populates it.

## Goals / Non-Goals

**Goals**

- A new `@unveiled/api` Bun workspace package built on Hono 4 that owns every HTTP route currently under `src/pages/api/**`.
- The canonical HTTP shape of every Astro Action under `src/actions/index.ts` (handler logic unchanged; only the binding moves).
- A deployable Cloudflare Worker (`packages/api/dist/worker.js`) consumed by `wrangler.api.toml`.
- An Hono-derived OpenAPI 3.1 document under `packages/api/openapi.generated.yaml`, kept in sync with TypeSpec via a new `openapi:check` step.
- A non-breaking shim so the existing `src/pages/api/**` endpoints keep working byte-for-byte during the transition window (the shim is removed in change 03).

**Non-Goals**

- Replacing Astro Actions as the mutation surface from the Astro app. (Handlers stay where they are; only the canonical HTTP shape is mirrored.)
- Moving SSR pages into Hono.
- Owning Astro-only data access from `src/lib/data-access/**`. The Astro app keeps its re-export shim until change 04.
- Cutting over the route boundary from Astro endpoints to the API worker. That is change 03.

## Decisions

### D1. Hono 4 as the HTTP runtime

Hono is small, Workers-native, and pairs with `@hono/zod-openapi` so the OpenAPI document can be assembled from the same Zod validators that parse requests. Alternatives considered:

- **Workers router types (itty-router).** Smaller surface area, but no first-class Zod/OpenAPI integration. We'd hand-roll request parsing against `src/lib/generated/request-validators.ts`.
- **tRPC.** Adds a different contract model than the OpenAPI document we already publish; would conflict with the TypeSpec source of truth.
- **Keep Astro endpoints.** Defers the three blockers listed above.

Decision: Hono + `@hono/zod-openapi`.

### D2. Single `src/pages/api/[...path].ts` shim, not per-file rewrites

The iteration note proposed rewriting each `*.ts` endpoint as a one-line re-export. That spreads the binding across many files and makes the cut-over in change 03 riskier (every file needs an edit). Instead, a single catch-all `src/pages/api/[...path].ts` calls `app.fetch(request, env)` and forwards to the Hono app. During the shim window, every existing path resolves to the Hono handler with zero behavior change. When change 03 ships the orchestrator Worker, this shim file is deleted in one place.

### D3. Shared handler functions, not duplicated route bodies

Both the Hono package and the (removed-in-change-03) shim call into shared handlers under `packages/api/src/handlers/**`. The handlers are framework-agnostic: they take a parsed input and return a typed result. Hono route files only handle Zod validation, auth context, and response shaping. This keeps handler logic identical across the transition and lets integration tests snapshot responses without going through HTTP.

### D4. Better Auth mounted inside `@unveiled/api`

Better Auth's HTTP handlers (`/api/auth/sign-up`, `/api/auth/sign-in`, `/api/auth/sign-out`, `/api/auth/forget-password`, etc.) live inside the Hono app at `/api/auth/*`. The Astro app keeps its session-cookie verification for SSR pages but defers all sign-in / sign-up / account endpoints to the API package. Cookie domain is configured via `AUTH_COOKIE_DOMAIN` so the session cookie is readable from both runtimes (Astro SSR pages and the API Worker). The auth handler module is mounted once (`app.route("/api/auth", authHandler)`) so the Astro app's `src/lib/auth.ts` continues to expose Better Auth for SSR page handlers.

### D5. Stripe webhook uses `stripe.webhooks.constructEventAsync`

The Stripe webhook handler moves into `packages/api/src/routes/stripe/webhook.ts`. Hono exposes `c.req.raw.body` as a `ReadableStream`; the Astro endpoint used `await request.text()`. We read the raw body once via `await c.req.raw.text()` and pass it to `stripe.webhooks.constructEventAsync` exactly like the Astro endpoint did. Behavior is preserved bit-for-bit.

### D6. OpenAPI document is generated by Hono, then diffed against TypeSpec

`@hono/zod-openapi` registers every route with a Zod request/response pair. `packages/api/src/openapi.ts` calls `app.openAPIGenerator` to produce the document and writes it to `packages/api/openapi.generated.yaml`. `bun run openapi:check` diffs this against `typespec/output/openapi.yaml` (modulo `servers`, `info.version`, and timestamps). Drift fails `bun run check`. This is additive: TypeSpec stays the source of truth, and the Hono document proves the runtime shape matches the contract.

### D7. Server-only deps move into `@unveiled/api`

`@neondatabase/serverless`, `stripe`, and the server-only `drizzle-orm` surface move from the root `devDependencies` into the package's `dependencies`. The root keeps `stripe` as a dev-dep until change 04 (Astro Actions still construct Stripe Checkout sessions). The package re-exports the Zod validators from `src/lib/generated/**` so its route files don't reach into app code.

### D8. Worker build is esbuild, not Wrangler's default bundler

`packages/api/package.json` runs `esbuild src/worker.ts --bundle --format=esm --target=es2022 --outfile=dist/worker.js` in `build`. `wrangler.api.toml` consumes the artifact at `packages/api/dist/worker.js`. esbuild is already in the toolchain (Hono's docs assume it; Astro's adapter also uses it). Wrangler's default bundler works but produces noisier output and slows iteration.

## Risks / Trade-offs

- **Two HTTP surfaces during the transition window.** → Mitigation: shared handler functions (`packages/api/src/handlers/**`) imported by both runtimes; integration tests under `tests/api/` snapshot responses for every existing path.
- **Hono on Workers + node-compat.** → Carried forward from `wrangler.toml` (`nodejs_compat`). Documented in `wrangler.api.toml`. Validated by `bun run preview:cloudflare` (the umbrella command runs `wrangler dev --remote` against the new config too).
- **Better Auth session-cookie domain mismatch.** → Mitigation: `AUTH_COOKIE_DOMAIN` is read by both the Astro app and the API worker from the shared env surface (`src/env.d.ts`). Documented in `design.md` and verified by an integration test that logs in via the Astro app, then makes an authenticated request to the API worker.
- **Webhook raw body.** → Mitigation: read once via `await c.req.raw.text()` and pass to `stripe.webhooks.constructEventAsync`. Existing Stripe webhook integration tests under `tests/e2e/` cover the behavior end-to-end.
- **OpenAPI drift between TypeSpec and Hono.** → Mitigation: `openapi:check` is wired into `bun run check` and runs both diffs in one step. Any drift fails CI before merge.
- **Bun workspace script fan-out.** → Mitigation: scripts that read from the repo root (e.g. `check`, `lint`, `specs:check`) widen their globs to cover `packages/api/**`; per-package scripts (`typecheck`, `test:unit`) fan out via `bun run --filter '*' ...` (existing pattern from change 01).

## Migration Plan

This change is non-breaking. Migration is a single deploy:

1. Merge the change → `packages/api/` ships; `src/pages/api/[...path].ts` shim is wired.
2. Deploy the new Worker via `bun run deploy:cloudflare:api` (added in this change) — runs `wrangler deploy -c wrangler.api.toml`.
3. The Astro deploy is unchanged; the catch-all shim makes the new Worker transparent during the transition.
4. Once the orchestrator Worker (change 03) ships, the catch-all shim is deleted in one PR.

Rollback: revert the merge; the Astro endpoints still work because `packages/api/` ships behind the catch-all shim and the Astro endpoints can be restored from git.

## Open Questions

- Do we want the API worker and the Astro app to share the same KV namespace id, or use distinct namespaces bound to the same underlying bucket? (Default: distinct namespaces so the Astro app can roll back independently.) Resolve before change 03's orchestrator PR.
- Should `/api/openapi.json` be served by the Hono app, or continue to be served by the Astro endpoint during the shim window? (Default: Hono app, reached via the catch-all shim.)
