# AGENTS.md

Canonical entrypoint for every contributor (human or AI agent) to the
**Unveiled MVP** codebase. Read this file first, end to end, before touching
any other doc. It is intentionally short (≤ 400 lines) and links out to the
deeper docs and specs that own each topic.

If you are a human contributor, also read [`CONTRIBUTING.md`](./CONTRIBUTING.md)
for the PR / review / release specifics that do not apply to agents.

## 1. What this is

Unveiled is a discovery, booking, and member-management web app for live
events. The migrated codebase (this repo) replaces the legacy app referenced
from `_old_app/`. The legacy tree is **read-only reference** — do not import
from it, do not copy logic out of it, and do not depend on its behavior.

**In scope:** Astro 6 SSR app, React 19 islands, Drizzle ORM data layer,
Better Auth, Stripe billing, Leaflet discovery map, gherkin/Playwright
parity coverage.

**Out of scope:** the legacy stack, any new microservice, anything not
captured by a capability spec in `openspec/specs/`.

## 2. Tech stack

Pinned to the versions in `package.json`; do not bump without an
architecture change.

- **Runtime / framework:** Astro 6 SSR on the Cloudflare Workers adapter
  (`@astrojs/cloudflare`), Node ≥ 22.12.
- **Client islands:** React 19 + `@astrojs/react` 5.
- **Package manager:** Bun (uses `bun.lock`). Use `bun` for install, scripts,
  and `bunx` for one-off CLIs. The root `package.json` declares
  `"workspaces": ["packages/*"]`; all dependencies stay hoisted at the
  root and per-package `dependencies` start empty until each workspace
  member is populated (see [`packages/README.md`](./packages/README.md)).
- **Database:** Drizzle ORM 0.45. Local dev uses PGlite (stored at
  `./.data/pglite`); cloud uses Neon Postgres via `@neondatabase/serverless`.
  Migrations live in `packages/app/drizzle/` and are generated by
  `drizzle-kit` invoked from the `@unveiled/app` package
  (`bun --filter @unveiled/app run db:generate`).
- **Auth:** Better Auth 1.6, mounted at `/api/auth/[...all]`. Session cookie
  is the source of truth for viewer identity.
- **Public landing surface:** `@unveiled/landing` is an Astro 6 SSR app
  mounted at `/*` in production (orchestrator in change 06). It reuses
  `@unveiled/design-system` primitives only — no direct HeroUI/NextUI
  dependency. Locally it runs on port 4322; the app runs on 4321; the
  orchestrator dev proxy listens on 4320.
- **Validation:** Zod 4 (pinned `4.3.6`). All action input/output goes
  through Zod schemas; do not hand-roll validators.
- **Styling:** Tailwind CSS v4 via `@tailwindcss/vite`, plus Hero UI
  component library. Design tokens are generated from
  `design-tokens.json` via Style Dictionary (`bun run tokens:gen`). The
  Hero UI library (`@nextui-org/react`) is a production dependency; the
  production primitives in `src/components/ui/` compose HeroUI as their
  base. The Ladle-only design replica under
  `src/components/ui/heroui-replica/` MUST NOT be imported by production
  code (gate: `bun run heroui-design-system-replica:check` and the
  permanent `bun run test:unit` import-graph guard in
  `tests/unit/no-ladle-replica-in-production.test.ts`).
- **Lint / format:** Biome (CLI version reported by `bunx biome --version`, currently 2.5.0) — see `biome.json`. CI runs `bun run check`,
  which fans out per-package (e.g. `astro check` in `@unveiled/app`,
  `biome check .` across the repo) and combines with `bun run specs:check`,
  and `bun run tokens:check`.
- **Server data:** TanStack Query 5 on the client; typed server actions in
  `src/actions/index.ts`; HTTP endpoints under `src/pages/api/`.
- **Billing:** Stripe (`@stripe/stripe-js` on the client, `stripe` on the
  server).
- **Discovery map:** Leaflet (open-source, no provider key required by
  default). Safe-public env values are namespaced `PUBLIC_*`.
- **Storage / deploy:** Cloudflare Workers + Pages for SSR, Cloudflare R2
  for media, Wrangler for deploys. Email is sent through Resend.
- **Contract:** TypeSpec (`typespec/main.tsp`) is the source of truth for
  every HTTP route and Astro Action. Generated artifacts (`typespec/output/`
  + `src/lib/generated/`) are committed. The OpenAPI document is served at
  `/api/openapi.json`.
- **Architecture:** LikeC4 model under `architecture/` is the source of
  truth for the C4 diagrams. Hand-edited Mermaid outside that directory is
  forbidden.
- **Feature tests:** Gherkin feature files under `tests/features/<domain>/`
  drive Playwright parity coverage via `bun run test:e2e`. Each per-feature
  folder at `tests/features/<domain>/<surface>/` holds the gherkin
  `feature.feature` plus a co-located `<component>.ladle.tsx` Ladle harness
  referenced by the scenarios' `@ladle(component=…, story=…)` tags. The
  OpenSpec change (`openspec/changes/<change-name>/`) is the single source
  of truth for the umbrella's `proposal.md` + `tasks.md`; per-feature
  proposals, task lists, and `specs.md` are not duplicated under
  `tests/features/`.

## 4. Coding conventions

The full rules live in [`docs/guidelines.md`](./docs/guidelines.md); the
short version:

- **TypeScript strict.** `tsconfig.json` extends `tsconfig.base.json`
  (which extends `astro/tsconfigs/strict`). No `any` without an inline
  justification comment. No `// @ts-ignore` or `// @ts-nocheck` without
  a justification comment.
- **Naming.** `kebab-case` for directories, scripts, and non-component files
  (`query-layer.ts`). `kebab-case` for Astro components
  (`event-card.astro`). `PascalCase` for React components/islands
  (`BookingModal.tsx`).
- **Imports.** Use the `@/` alias for everything under `src/` (legacy
  compatibility shim; removed in change 04). Cross-package imports MUST
  use the `@unveiled/<pkg>` alias declared in the root `tsconfig.base.json`.
  Cross-package relative imports are forbidden.
- **Formatting.** Biome enforces double quotes, trailing commas, semicolons,
  and 2-space indent. Run `bun run format` before committing.
- **No comments unless asked.** The codebase policy is no inline comments
  unless a reviewer explicitly asks for one. Self-documenting names preferred.
  The single permitted exception is the `// @ladle-only` header on every file
  under `src/components/ui/heroui-replica/`, which is enforced by
  `bun run heroui-design-system-replica:check`.
- **Server actions are the only mutation surface.** Page-level forms call
  actions in `src/actions/index.ts`; they do not call HTTP endpoints directly.
- **No `_old_app/` imports.** If a legacy file is the only source of an
  algorithm, port it into `src/` and reference the new module.

## 5. OpenSpec workflow

This repo is governed by [OpenSpec](https://openspec.dev). Capability specs
are the contract for "what the system should do"; change proposals are the
contract for "how we are changing it."

- **Read the spec first.** Before writing a proposal, skim
  [`openspec/specs/`](./openspec/specs/) to find the capability your change
  touches. Each capability lives in its own folder with a `spec.md` that lists
  its `### Requirement:` blocks and `#### Scenario:` blocks.
- **Pick a change name.** Kebab-case, e.g. `add-venue-checkin`. The full
  flow is automated by `openspec new change <name>` and validated by
  `openspec validate <name>`.
- **For every change, write four artifacts in order:**
  1. `proposal.md` — *why* (problem, opportunity) and *what* (capabilities
     added/modified). Lists every capability the change will modify.
  2. `specs/**/*.md` — one file per capability: `## ADDED Requirements` for
     new caps, `## MODIFIED Requirements` for existing caps (paste the full
     updated requirement block under the header), `## REMOVED Requirements`
     with a `**Reason**` and `**Migration**` for deletions. Every
     `### Requirement:` MUST have at least one `#### Scenario:` (WHEN/THEN).
  3. `design.md` — *how* (context, goals/non-goals, decisions, risks,
     migration). Only for cross-cutting changes, new external deps, security
     or performance work, or anything ambiguous.
  4. `tasks.md` — implementation checklist using `- [ ] X.Y` checkboxes
     grouped under `## N.` headings, ordered by dependency. The apply phase
     parses this format to track progress.
- **Capabilities drive the contract.** A capability listed in `proposal.md`
  *MUST* have a matching delta in `specs/`. Reviewers will reject the
  proposal if a capability is named in one artifact and not the other.
- **Validate locally** with `openspec validate <name>` (or the project
  equivalent wired into `bun run check`). Resolve every error before opening
  the PR.
- **Archive after merging.** Once the change lands, run
  `openspec archive <name>`. The proposal moves to
  `openspec/changes/archive/<date>-<name>/` and its spec deltas fold into
  the live `openspec/specs/`.

## 6. Iteration cycle

The repo follows a numbered iteration plan under
[`.development-plan/0N-iteration/`](./.development-plan/) — the maintainer's
private planning space. **Agents MUST NOT modify or rely on it**; the
authoritative artifacts for any change are always under `openspec/`. The
plan exists so the maintainer can stage multiple specs across cycles; the
public contract is the OpenSpec change.

For every change, an agent should:

1. Read the active change's `openspec/changes/<change-name>/proposal.md`
   and `tasks.md` top to bottom.
2. Implement the change, mark each checkbox as it completes, and update any
   dependency specs (`openspec/specs/<capability>/spec.md`) as you go.
3. Add per-feature BDD specs under
   `tests/features/<domain>/<surface>/{feature.feature, <component>.ladle.tsx}`.
4. Run the full check (`bun run check`) and any test/gherkin commands named
   in the proposal.
5. Update `AGENTS.md` if the stack, file layout, toolchain commands, or
   definition of done changed.
6. Hand off to a human reviewer with a green CI; do not self-merge.

## 8. Definition of done

A change is *done* only when all of the following are true:

- [ ] Every task in the change's `tasks.md` is checked off.
- [ ] `packages/orchestrator/` builds and produces a deployable Worker bundle (`bun run build` writes `packages/orchestrator/dist/worker.js`).
- [ ] `bun run check` passes locally.
- [ ] `bun run test:e2e` passes (gherkin parity suite against the orchestrator's port-4320 proxy).
- [ ] `bun run test:ladle` passes for any feature spec that adds a new component.
- [ ] `bun run ladle:coverage` shows no drift.
- [ ] `bun run specs:check` shows no drift (TypeSpec artifacts in sync; `servers` block points at the orchestrator's hostname).
- [ ] `bun run arch:check` shows no drift (LikeC4 model in sync; orchestrator is the entry container with three downstream service bindings).
- [ ] `bun run tokens:check` shows no drift (design tokens in sync).
- [ ] `GET /healthz` returns `200` with body `ok`; `GET /readyz` returns `200` only when all downstream Workers' readiness probes are green.
- [ ] `bun run dev` boots all four Workers behind a single local port (4320) with no port conflicts.
- [ ] No file outside `packages/design-system/**` imports `@nextui-org/*` or `@heroui/*` (enforced by `tests/unit/design-system-hero-ui-boundary.test.ts`, which is part of `bun run test:unit` and therefore `bun run check`).
- [ ] `bun run deploy:cloudflare` deploys in dependency order (api → app → landing → orchestrator) and the production hostname serves every URL prefix correctly.
- [ ] The OpenSpec proposal is either updated in place, or a new proposal
      is added that supersedes it, and `openspec validate` passes.
- [ ] `AGENTS.md` is updated if the stack, file layout, toolchain
      commands, or definition of done changed.
- [ ] The change is archived (`openspec archive <name>`) once the PR merges.

## 9. What NOT to do

- **No imports from `_old_app/`.** Reference the legacy tree by reading; do
  not couple code to it.
- **No hand-edited Mermaid.** All C4 diagrams are generated from the LikeC4
  model under `architecture/`.
- **No hand-edited generated files.** Do not edit anything under
  `typespec/output/` or `src/lib/generated/`. Change the TypeSpec source
  and re-run `bun run specs:gen`.
- **No `getByText` chains in gherkin.** Tests are selector-driven (see the
  gherkin spec for the selector discipline).
- **No `data-testid` for production behavior.** `data-testid` is for tests
  only and must not gate production logic.
- **No commits to `main` without a green `bun run check`.**
- **No new capability without a spec.** Anything user-visible must show up
  as a `### Requirement:` + `#### Scenario:` in a capability spec.
- **No comments in code unless asked.** Follow the no-comments policy.
